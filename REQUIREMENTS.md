# Diary — Functional Requirements

## 1. Overview

Diary is a local, single-user, AI-powered daily task management app for macOS. It runs as a background server with multiple client interfaces: a TUI for full task management, a conversational AI chat mode, and a machine-friendly CLI API for external tool integration.

The core philosophy is **minimal friction**. Every design decision optimizes for the least effort required to keep tasks organized. The AI handles the heavy lifting — the user just talks to it in plain text.

- **Language**: Python
- **AI framework**: Strands Agents
- **TUI framework**: Textual
- **Platform**: macOS (v1)
- **Storage**: Local only, single user, no network required (except for AI model API calls)

## 2. Task Model

### 2.1 Task

| Field          | Required | Description                                                                                      |
|----------------|----------|--------------------------------------------------------------------------------------------------|
| id             | auto     | Unique identifier, auto-generated                                                                |
| title          | yes      | Short, descriptive name                                                                          |
| description    | no       | Free-form context about the task                                                                 |
| status         | yes      | One of: `todo`, `in-progress`, `done`. Default: `todo`                                           |
| priority       | no       | One of: `high`, `medium`, `low`. Default: `medium`                                               |
| tags           | no       | List of hashtag-style labels (e.g., `#backend`, `#urgent`)                                       |
| due_date       | no       | Optional deadline                                                                                |
| scheduled_date | no       | The date this task is planned to be worked on. `null` = backlog.                                  |
| archived       | auto     | Whether the task is archived. Default: `false`                                                   |
| archive_reason | no       | Required when archiving. Free-form string explaining why.                                        |
| created_at     | auto     | Timestamp of creation                                                                            |
| updated_at     | auto     | Timestamp of last modification                                                                   |

### 2.2 Activity Log Entry

Each task has an ordered list of activity log entries.

| Field     | Description                                                                                         |
|-----------|-----------------------------------------------------------------------------------------------------|
| id        | Unique identifier for the entry (needed for edit/delete targeting)                                  |
| timestamp | When the entry was created                                                                          |
| source    | `user`, `ai`, or `system`                                                                           |
| content   | Free-form text, kept concise and to the point                                                       |

**Source types:**

- **user**: Manually written by the user (via TUI or AI chat). E.g., "Got stuck on token refresh."
- **ai**: Generated by the AI during conversation. Clearly labeled. E.g., "[AI] Created from chat: user mentioned needing to fix login bug."
- **system**: Auto-logged mechanical changes. E.g., "Priority changed from low to high", "Status changed to done", "Scheduled for 2026-03-02."

### 2.3 Tags

- Free-form hashtag-style labels. E.g., `#backend`, `#meeting`, `#p0-fix`.
- Used for filtering.
- When the AI creates a new task, it looks at existing tags across all tasks and suggests relevant ones. The user can accept, modify, or reject.

## 3. Task Lifecycle

```
                ┌──────────┐
    ┌──────────►│ Backlog  │◄──────────┐
    │           │ (todo,   │           │
    │           │ no date) │           │
    │           └────┬─────┘           │
    │                │                 │
    │         schedule for             │
    │           a date                 │
    │                │                 │
    │           ┌────▼─────┐           │
    │           │  Today   │     move back to
    │           │(todo /   │       backlog
    │           │in-progress)──────────┘
    │           └────┬─────┘
    │                │
    │           mark done
    │                │
    │           ┌────▼─────┐
    └───────────│   Done   │
     reopen     │  (done)  │
                └────┬─────┘
                     │
                  archive
                     │
                ┌────▼─────┐
                │ Archived │
                │(+ reason)│
                └────┬─────┘
                     │
                 un-archive
                 (→ backlog)
                     │
                     └──────────► back to Backlog
```

- **Backlog**: `status=todo` and `scheduled_date=null`. Exists but not planned for any specific day.
- **Today**: `scheduled_date=today`. Can be `todo` (not started) or `in-progress`.
- **Done**: `status=done`. Retains `scheduled_date` for history.
- **Archived**: `archived=true`. Disappears from all active views. Retains all data and activity logs. Requires a reason string. Can be un-archived, which returns it to backlog.

**There is no automatic status transition.** The user (or AI on user's behalf with explicit confirmation) always decides.

### 3.1 Work Periods

A task can move in and out of active work across multiple days. Work periods are **derived data** computed from the activity log's system entries (schedule/backlog/done transitions). No extra fields needed.

Example:
```
Task: "Auth refactor"
Work periods (computed from activity log):
  Mar 2–4  (scheduled → moved to backlog)
  Mar 10–11 (scheduled → marked done)
```

If system log entries are deleted by the user, work period computation reflects the remaining data (best effort).

## 4. Architecture

### 4.1 Shared Service Layer (No Server)

- No background server or daemon process.
- All clients (TUI, chat, API, zsh hook) share a Python service layer that reads/writes SQLite directly.
- SQLite in WAL mode handles concurrent access from multiple clients.
- The database file is the single source of truth.

### 4.2 Notifications via launchd

- Two macOS `launchd` calendar jobs (morning and evening) run one-shot notification scripts.
- Each script sends a hardcoded macOS notification via `terminal-notifier` and exits immediately.
- Clicking the notification brings Warp to the foreground. The user types `diary tui` or `diary chat`.

### 4.3 Clients

All clients import the shared service layer directly. No HTTP, no IPC.

## 5. Client: TUI

Launched via `diary tui`. Built with Textual.

### 5.1 Views

- **Today panel**: Tasks with `scheduled_date=today`, sorted by priority (high → medium → low). Shows title, priority, status.
- **Backlog panel**: Tasks with `scheduled_date=null`, sorted by priority.
- **Task detail view**: Selected task's full fields and activity log.
- **AI chat panel**: Embedded conversational AI section. Conversation history retained for the TUI session duration.
- **History view**: Separate screen showing completed tasks organized by date (calendar-style), navigable by day/week.
- **Archive view**: Separate screen showing archived tasks with their archive reasons.

### 5.2 Manual Task Operations (TUI only)

The user can perform all of the following directly in the TUI without the AI:

- Create a task
- Edit any task field (title, description, status, priority, tags, due date, scheduled date)
- Add activity log entries to a task
- Edit activity log entry text (TUI only — not available via AI or API)
- Delete activity log entries, including system entries (TUI only — not available via AI or API)
- Schedule a task for today or another date
- Move a task to backlog (clear scheduled date)
- Mark a task as done
- Reopen a done task
- Archive a task (with required reason)
- Un-archive a task (returns to backlog)
- **Manual refresh** via keyboard shortcut to reload data from the server (handles external changes made via `diary api` or another client)

## 6. Client: Conversational Chat CLI

Launched via `diary chat`. Opens an interactive conversational session with the AI in the current terminal.

- No TUI panels — just back-and-forth text conversation.
- Same AI behavior and capabilities as the TUI's chat panel.
- Conversation history retained for the session duration.

### 6.1 What the AI Cannot Do

- Delete activity log entries
- Edit activity log entry text

These are restricted to TUI manual controls only.

---

## 7. Client: Machine CLI API

Non-interactive, deterministic commands for external tool integration. Designed for other AIs (coding assistants, etc.) to call as subprocess commands.

- Returns structured JSON output.
- Every command has a `--help` option for discoverability.

### 7.1 Commands

**List tasks (with filters):**
```
diary api list [filters]
  --scheduled today|<date>|none    # none = backlog
  --status todo|in-progress|done
  --priority high|medium|low
  --tag <tag>
  --from <date> --to <date>        # date range filter
  --search <text>                  # full-text search across title and description
  --archived                       # include archived tasks
  --verbose                        # full detail including activity log
```

Default output (non-verbose): `id`, `title`, `status`, `priority`, `tags`.
Verbose output: all fields + full activity log.

All filters are combinable.

**Show single task:**
```
diary api show <task-id>
```
Returns full task detail including activity log.

**Create task:**
```
diary api add --title <title> [--description <desc>] [--priority high|medium|low] [--tags <comma-separated>] [--due <date>] [--schedule <date>|today]
```

**Update task:**
```
diary api update <task-id> [--title <title>] [--description <desc>] [--status todo|in-progress|done] [--priority high|medium|low] [--tags <comma-separated>] [--due <date>] [--schedule <date>|today|none]
```
`--schedule none` moves to backlog.

**Add activity log entry:**
```
diary api log <task-id> <message>
```
Logged with `source=user`.

**Archive / un-archive:**
```
diary api archive <task-id> --reason <reason>
diary api unarchive <task-id>
```

### 7.2 Restrictions

The machine CLI API **cannot**:
- Delete activity log entries
- Edit activity log entries

All mutations through the API are recorded as system entries in the activity log.

## 8. AI Behavior

### 8.1 Technology

Built with Strands Agents (Python). The AI is a tool-using agent with access to all task operations via the server API (except activity log deletion/editing).

- **Model**: Amazon Bedrock, default Claude Sonnet (configurable via config file).
- The model name/ID is a config setting so the user can switch models without code changes.

### 8.2 Conversational Principles

1. **Always conversational.** The AI never silently makes changes. Every interaction is a back-and-forth dialogue.
2. **Transparent.** Before any mutation, the AI presents exactly what it will do — every field, every change — and waits for confirmation.
3. **Proactive field gathering.** When creating or updating a task, the AI asks about every optional field the user hasn't provided. The user can skip fields, but the AI must ask. It fills what it can infer, suggests what it can (e.g., tags), and asks about the rest.
4. **Free-form confirmation.** The user can respond with yes/no, but also with clarifications, corrections, or additional context. The AI adapts accordingly.
5. **Concise.** Short sentences, no fluff. Matches the user's casual tone.

### 8.3 Interaction Flow

```
User: I need to fix the login bug, it's pretty urgent

AI: I'll create a new task:
  Title: Fix login bug
  Priority: ? (you said urgent — should I set this to high?)
  Tags: I'd suggest #backend, #auth based on existing tags
  Scheduled: today or backlog?
  Due date: any deadline?
  Description: anything else to note?

User: high priority, today, no due date, that's it

AI: Creating:
  Title: Fix login bug
  Priority: high
  Tags: #backend, #auth
  Scheduled: today
  Due date: none
  Description: none
Confirm?

User: yes

AI: Done. "Fix login bug" created and scheduled for today.
```

### 8.4 Capabilities

Upon user request and confirmation, the AI can:

- **Create tasks** with auto-suggested tags based on existing tags in the system
- **Update any task field** (status, priority, tags, due date, scheduled date, title, description)
- **Add activity log entries** to tasks
- **Reschedule tasks** (move between today, another date, or backlog)
- **Summarize** a day, week, or arbitrary time range based on activity logs and task data
- **Suggest next task** to work on based on priority and pending items (when asked)
- **Bulk operations** (e.g., "move all low-priority tasks to backlog") with confirmation
- **Persist context** from conversation into task data (activity log or description), labeled as AI-generated. Should proactively ask "should I save this to the task?" when the user provides context that seems important.

### 8.5 What the AI Cannot Do

- Delete activity log entries
- Edit activity log entry text
- Make any change without explicit user confirmation

### 8.6 Context & Memory

- **No memory across sessions.** Each `diary chat` or TUI session starts fresh.
- The AI's context is the task data itself (read from the server on each interaction).
- If the user provides context that should be remembered, the AI persists it into the relevant task's activity log or description, clearly labeled as AI-generated (`source=ai`).

## 9. Notifications

Managed by the background server via `launchd` scheduling. Uses macOS native notifications (`terminal-notifier` or `osascript`).

### 9.1 Morning Notification

- Fired by a macOS `launchd` calendar job at a configured time (default: 10:00 AM) on workdays.
- Sends a hardcoded macOS notification: "Time to check your tasks" (no DB access, no task count).
- Clicking the notification brings Warp to the foreground.

### 9.2 Evening Notification

- Fired by a macOS `launchd` calendar job at a configured time (default: 6:00 PM) on workdays.
- Sends a hardcoded macOS notification: "Time to review your tasks."
- Clicking the notification brings Warp to the foreground.

---

## 10. Zsh Integration

A snippet added to `.zshrc` that runs on every new terminal session.

On new terminal open, it calls the server and prints a brief summary:

```
📋 Diary — 3 tasks planned for today:
  1. [HIGH] Auth refactor
  2. [MED]  DB migration
  3. [LOW]  Update docs

Run `diary chat` to review or `diary tui` for full view.
```

- Shows task titles and priorities only. No descriptions, no activity logs.
- If no tasks are scheduled for today, says so and suggests reviewing the backlog.
- Prints in every new terminal session (no deduplication).

## 11. Configuration

Stored in a config file (e.g., `~/.config/diary/config.toml`).

| Setting                  | Default               | Description                          |
|--------------------------|-----------------------|--------------------------------------|
| morning_notification_time| 10:00                 | Time for morning notification        |
| evening_notification_time| 18:00                 | Time for evening notification        |
| workdays                 | Mon–Fri               | Days notifications are active        |
| notifications_enabled    | true                  | Enable/disable all notifications     |
| data_dir                 | ~/.local/share/diary  | Where the database is stored         |
| ai_model                 | bedrock/claude-sonnet | Bedrock model name/ID                |
| terminal_app             | Warp                  | Terminal app for notification click   |

## 13. V1 Scope

What must work for v1:

1. Background server with task CRUD API
2. TUI with today panel, backlog panel, task detail view, AI chat panel
3. `diary chat` conversational CLI
4. `diary api` machine-friendly CLI (list with filters, show, add, update, log, archive, unarchive)
5. Zsh integration for new terminal summary
6. Morning and evening macOS notifications (clicking opens TUI)
7. AI agent (Strands) that can create, update, reschedule, log, and summarize tasks via conversation
8. Proactive field gathering and tag suggestion by AI
9. Activity log with user/ai/system entries
10. Activity log edit and delete via TUI manual controls only
11. Task archiving with reason (soft delete)
12. Config file for notification times, workdays, server settings, and AI model
13. Full-text search across task titles and descriptions

---

## 14. Out of Scope for V1

- Cloud sync / multi-device
- Multi-user / sharing
- Export (markdown, PDF, etc.)
- Linux / Windows support
- Web UI
- Recurring tasks
- Time tracking
- Integrations with external tools (Jira, GitHub, etc.)
- Manual task reordering in today's list
